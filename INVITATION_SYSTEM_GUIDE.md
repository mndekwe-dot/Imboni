## 1. Settings & Environment Variables

### What this does

The `.env` file stores values that can change between environments (development, production) without touching the code.

`INVITATION_EXPIRY_DAYS` is read by `settings.py` using `config()` from `python-decouple` and cast to an integer. This value is used in views whenever an invitation is created — it controls how many days until the link stops working.

The `AFRICASTALKING_*` variables are credentials for the Africa's Talking SMS/WhatsApp API. `AFRICASTALKING_USERNAME` is `sandbox` during development (same idea as Mailtrap — a test environment where no real SMS is sent). `AFRICASTALKING_SENDER_ID` is the name shown as the sender on SMS messages.

In `settings.py`, these values are loaded with `config()` and `cast=int` so Django treats the number as an integer not a string.

`FRONTEND_URL` is stored here because invitation links point to the React frontend. When the backend builds a link like `/register/uid/token/`, it needs to know the frontend's base URL. In development this is `http://localhost:3000`. In production it becomes your real domain.

## 2. Install Dependencies

### What this does

`africastalking` is a Python SDK (Software Development Kit) provided by Africa's Talking. An SDK is a pre-built library that wraps their API so you don't have to write raw HTTP requests. You call `sms.send(message, [phone])` and the SDK handles the rest.

It is added to `requirements.txt` so anyone else setting up the project can run `pip install -r requirements.txt` to get all packages at once.

## 3. Invitation Model

### File: `Backend/authentication/models.py`

### What this does

The `Invitation` model is a database table. Every time an admin or DOS invites someone, one row is created in this table. The row tracks everything about that invitation from creation to completion.

### Field Explanations

| Field | Type | Why it exists |
|---|---|---|
| `id` | UUID | Uses a random ID instead of 1, 2, 3. This makes it impossible to guess other invitation IDs |
| `email` | EmailField (optional) | Where the invitation email is sent. Optional because some people only have a phone number |
| `phone_number` | CharField (optional) | Used for SMS and WhatsApp. Optional because some people only have email |
| `first_name / last_name` | CharField | Needed to address the person by name in the invitation message |
| `role` | CharField (choices) | What role this person will receive when they register. Set by the inviter |
| `invited_by` | ForeignKey to User | Tracks who sent the invitation. Uses `SET_NULL` so if the inviter is deleted, the invitation record is kept |
| `token` | CharField (unique) | A secure random string generated by Django. Part of the invitation link URL. unique=True prevents two invitations sharing the same token |
| `uid` | CharField | The invitation's UUID encoded into a URL-safe format using `urlsafe_base64_encode`. Also part of the link URL |
| `is_used` | BooleanField | Becomes `True` after someone registers with this link. Prevents the same link being used twice |
| `expires_at` | DateTimeField | The deadline. After this datetime, the link no longer works |
| `channels_sent` | JSONField | Stores which channels succeeded, e.g. `["email", "sms"]`. Useful for admin tracking |
| `delivery_status` | CharField (choices) | Overall status: `pending`, `sent`, or `failed` |
| `student` | ForeignKey (optional) | Only used when inviting a parent — links them to a specific student from the start |
| `created_at` | DateTimeField (auto) | Automatically set when the row is created |

### Property Explanations

`is_expired` is a Python **property** — it behaves like a field but is calculated on the fly. It compares the current time (`timezone.now()`) against `expires_at`. If the current time is past the deadline, it returns `True`.

`is_valid` combines both checks: an invitation is only valid if it has NOT been used AND has NOT expired. This is called in `CompleteRegistrationView` as a single clean check before creating an account.

### Why UUID as primary key

Django normally uses auto-incrementing integers (1, 2, 3...) as IDs. With integers, someone could try `/register/1/token/`, `/register/2/token/` and guess IDs. A UUID like `dab53ee5-f0e6-42e8-8fda-253923...` has 2^122 possible values — impossible to guess.

## 4. Migration

### What this does

A **migration** is a Python file Django generates that describes a database change. When you run `makemigrations`, Django reads your models and writes a migration file. When you run `migrate`, Django applies those changes to the actual database — creating tables, adding columns, etc.

The `pending_email` field was added to the `User` model to support the email change feature. It stores the new email address while waiting for the user to confirm it. The user's real `email` field stays unchanged until the confirmation link is clicked. This prevents someone from losing access to their account if they accidentally type the wrong new email.

## 5. Permissions

### File: `Backend/authentication/permissions.py`

### What this does

`CanInvite` is a custom DRF (Django REST Framework) permission class. DRF checks permissions before a view runs. If the permission check fails, the view never executes and a `403 Forbidden` response is returned automatically.

### How it works

`INVITE_PERMISSIONS` is a dictionary that defines the rules:
- `admin` can invite any role
- `dos` (Director of Studies) can only invite teachers and students
- `discipline` can only invite students and matrons

`has_permission` is called automatically by DRF on every request. It checks two things: the user is logged in, and their role exists as a key in `INVITE_PERMISSIONS`. If either fails, the request is blocked.

`can_invite_role` is called **manually inside the view** with a specific target role. Even if a DOS has permission to access the invite view, they still cannot invite an admin — this method checks that specific role pair.

### Why two separate checks

`has_permission` is a gate — it asks "can this user access invite views at all?". `can_invite_role` is a filter — it asks "can this specific user invite this specific role?". Separating them means the role check logic is reusable across `SendInvitationView` and `BulkInviteView`.

## 6. Serializers

### File: `Backend/authentication/serializers.py`

### What this does

Serializers in DRF do two things: they **validate** incoming data and they **transform** data going out (Python objects → JSON).

### InvitationSerializer

Defines what data is accepted when an admin creates an invitation. The `read_only_fields` list contains fields that are returned in the response (so the frontend can display them) but are **not accepted as input** — they are set by the backend. For example, `expires_at` is calculated in the view, not provided by the user.

The `validate()` method is a custom validation rule that runs after individual field validation. It checks that at least one contact channel exists. This is necessary because both `email` and `phone_number` are individually optional, but we need at least one to be able to send the invitation anywhere.

### CompleteRegistrationSerializer

Validates the data submitted when someone fills in the registration form. `write_only=True` on the password fields means they are accepted as input but never returned in any response — a security measure to ensure passwords never leak into API responses. `min_length=8` enforces a minimum password length at the serializer level before the data even reaches the view.

### EmailChangeRequestSerializer

Has a `validate_new_email` method — this is a **field-level validator**. DRF calls it automatically because it follows the naming convention `validate_<field_name>`. It queries the database to check the email is not already in use before the request proceeds.

## 7. Channel Service

### File: `Backend/authentication/service.py`

### What this does

This file is a **service layer** — code that handles a specific job (sending messages) separated from the views. Views should focus on HTTP request/response logic. The service focuses on communication logic. This separation makes each part easier to test and change independently.

### Function Explanations

**`initialize_africastalking()`**
Called before any SMS or WhatsApp send. Africa's Talking requires initialization with credentials before you can use their API. It is kept as a separate function because both `send_invitation_sms` and `send_invitation_whatsapp` need it.

**`send_invitation_email()`**
Renders the HTML email template using `render_to_string` — it fills the template variables (`first_name`, `role`, `registration_link`, `expiry_days`) with real values and returns the final HTML string. Then sends it using Django's `send_mail`. `fail_silently=False` means if the email server rejects the message, an exception is raised so `dispatch_invitation` can catch it and log it.

**`send_invitation_sms()`**
Uses Africa's Talking SMS service. The message is plain text (no HTML) because SMS does not support formatting. The phone number is passed as a list because the API supports bulk sending to multiple numbers.

**`send_invitation_whatsapp()`**
WhatsApp supports `\n` newlines for formatting. The message is structured to be readable on a phone screen with clear sections.

**`dispatch_invitation()`**
The main function that views call. It checks what contact information is available on the invitation and calls the matching send functions. Each send is wrapped in `try/except` so if one channel fails (e.g. phone number is wrong format), the others still proceed. It returns a list of channels that succeeded. This list is saved to `invitation.channels_sent` so you can track what was delivered.

## 8. Views

### File: `Backend/authentication/views.py`

### Imports at the top

`default_token_generator` — Django's built-in token generator. Produces a secure, time-limited token tied to a specific user's data. If the user's password changes, old tokens become invalid automatically.

`urlsafe_base64_encode / urlsafe_base64_decode` — converts bytes to a URL-safe string and back. Used to put the invitation's UUID into a URL without special characters breaking the URL.

`force_bytes` — converts a string or integer to bytes, which is required by `urlsafe_base64_encode`.

`timezone` — Django's timezone-aware datetime. Always use this instead of `datetime.now()` because the project uses `Africa/Kigali` timezone in settings.

---

### View 1 — SendInvitationView

**Endpoint:** `POST /imboni/auth/invite/`

This view creates a single invitation. The flow is:

1. Check if the requesting user's role can invite the target role (using `CanInvite`)
2. Validate the invitation data (using `InvitationSerializer`)
3. Save the invitation with placeholder values for `token` and `uid` — this is necessary because `expires_at` has a NOT NULL constraint in the database, so we must provide a value before saving
4. Generate a real token and uid from the saved invitation's UUID
5. Update the invitation record with the real token, uid, and expiry
6. Build the registration link and call `dispatch_invitation`
7. Save the channels sent and delivery status
8. Return the created invitation data

The registration link format is `FRONTEND_URL/register/uid/token/` — this is the URL that goes in the email. React receives it, extracts uid and token, and calls the verify endpoint.

---

### View 2 — BulkInviteView

**Endpoint:** `POST /imboni/auth/invite/bulk/`

Accepts a list of invitation objects and processes them one by one. Unlike `SendInvitationView`, it does not stop on failure — each invitation is tried independently and the result goes into either `results['sent']` or `results['failed']`. This way a single bad entry does not block the rest. The response contains both lists so the admin can see exactly what happened.

---

### View 3 — InvitationListView

**Endpoint:** `GET /imboni/auth/invite/list/`

Returns invitations filtered by role. Admins see all invitations in the system. Other authorized roles (DOS, discipline) only see invitations they personally sent. This uses `get_queryset()` which is the DRF way to dynamically filter the data returned by a list view.

---

### View 4 — ResendInvitationView

**Endpoint:** `POST /imboni/auth/invite/resend/<pk>/`

Generates a fresh token and extends the expiry. The old token becomes invalid as soon as the new one is saved (because `token` is unique and the old one is overwritten). The `dispatch_invitation` call sends to the same channels as before, because the email and phone_number on the invitation object have not changed.

---

### View 5 — CancelInvitationView

**Endpoint:** `DELETE /imboni/auth/invite/<pk>/cancel/`

Deletes the invitation from the database. First checks `is_used` — there is no point cancelling an invitation that has already been completed. Uses HTTP `DELETE` method which is the RESTful convention for deletion operations.

---

### View 6 — VerifyInvitationView

**Endpoint:** `GET /imboni/auth/register/verify/<uid>/<token>/`

Called by React when the invitee clicks the link in their email. It does not create anything — it just verifies the link is valid and returns the pre-filled invitation details (name, role, email) so the registration form can display them. Three checks are performed in order: is it used, is it expired, does the token match. Returning the invitation details saves the user from having to type their name again since it was set by the admin.

---

### View 7 — CompleteRegistrationView

**Endpoint:** `POST /imboni/auth/register/complete/`

The final step. Validates the registration data, re-verifies the invitation (the link could have expired between the verify step and form submission), checks the username is not taken, then creates the user account. The user's email, first name, last name, and role all come from the invitation — not from what the user types — so the admin fully controls what role is assigned. After creation, marks the invitation as used so the link cannot be used again, then sends the welcome message.

### _send_welcome_message function

This is a standalone function (not inside a class) because it is called from within `CompleteRegistrationView.post()` and needs to be defined at module level. It sends a welcome email using the `welcome_email.html` template. `fail_silently=True` means if the welcome email fails, the registration still succeeds — the account is already created at this point so it would be wrong to return an error.

---

### View 8 — EmailChangeRequestView

**Endpoint:** `POST /imboni/auth/email-change/request/`

Requires authentication (`IsAuthenticated`). Saves the new email to `pending_email` on the user, generates a token tied to that user, and sends a confirmation link to the **new** email address (not the old one). The old email stays active until confirmation.

---

### View 9 — EmailChangeConfirmView

**Endpoint:** `GET /imboni/auth/email-change/confirm/<uid>/<token>/`

Verifies the token, moves `pending_email` into `email`, clears `pending_email`, then notifies the **old** email address that the change happened. The notification to the old email is a security measure — if someone hijacks your account and changes your email, you will receive a warning at your old address.

## 9. URLs

### File: `Backend/authentication/urls.py`

### What this does

Each `path()` maps a URL pattern to a view class. `.as_view()` converts the class-based view into a callable that Django's URL dispatcher can call.

`<uuid:pk>` is a URL converter — it only matches UUID-formatted strings and converts them to a Python `uuid.UUID` object automatically. This prevents someone from passing random text as a pk.

`<str:uid>` and `<str:token>` accept any string. The uid and token are not UUIDs — uid is a base64-encoded UUID and token is a Django security token, so `<str:>` is the correct converter.

The URL names (e.g. `name='send-invitation'`) are used by `reverse()` and `{% url %}` in templates to generate URLs without hardcoding paths.

## 10. Email Templates

### Folder: `Backend/templates/emails/`

### What this does

These are HTML files that Django renders using its template engine. `{{ variable }}` syntax is replaced with real values when `render_to_string()` is called.

The templates are registered by setting `'DIRS': [os.path.join(BASE_DIR, 'templates')]` in `settings.py`. This tells Django where to look when `render_to_string('emails/invitation_email.html', {...})` is called.

### invitation_email.html

The invitation email sent when an admin creates an invitation. Uses `{{ first_name }}`, `{{ role }}`, `{{ registration_link }}`, and `{{ expiry_days }}` as template variables. The button links directly to the React registration page. A plain-text link fallback is included below the button for email clients that block images or buttons.

### welcome_email.html

Sent after successful registration. Uses `{{ first_name }}`, `{{ username }}`, `{{ role }}`, and `{{ login_link }}`. Green header to distinguish it visually from the blue invitation email. Shows the username because the user chose it themselves and may want a reminder.

### email_change_email.html

Used for the email change confirmation flow. Sent to the **new** email address asking the user to confirm the change.

### Why HTML templates instead of plain text

HTML emails display consistently across all mail clients (Gmail, Outlook, mobile). The button, colors, and layout make the email look professional and trustworthy. Plain text fallback is always included in the `message=` parameter of `send_mail()` for clients that cannot display HTML.

## Common Errors and Fixes

| Error | Cause | Fix |
|---|---|---|
| `africastalking not found` | Package not installed | `pip install africastalking` |
| `TemplateDoesNotExist` | Templates folder not found | Check `TEMPLATES DIRS` in settings.py |
| `Invitation matching query does not exist` | Wrong uid in URL | Check uid is being passed correctly |
| `Token invalid` | Token expired or already used | Resend invitation from admin |
| `IntegrityError on username` | Username already taken | Choose different username |
| `IntegrityError on expires_at` | Serializer saved before expires_at was set | Pass expires_at directly into serializer.save() |
| `NameError: send_mail` | send_mail not imported in views.py | Add send_mail to the import from django.core.mail |
